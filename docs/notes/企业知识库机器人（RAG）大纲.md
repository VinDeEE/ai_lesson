好，我们就做 **A：企业知识库机器人（RAG）**。我给你一个“能跑通→能变准→能上线”的落地路线：你照着做，2–3 天能出 MVP，2–3 周能做到可用级。

------

## 目标定义（你要做出来的东西）

输入：用户问题
输出：

1. **回答**（基于知识库）
2. **引用证据**（引用哪几段原文）
3. **置信度/命中情况**（高/中/低）
4. **需要人工**（当找不到证据或冲突时）

------

## 系统架构一张图（从头到尾）

1. 文档准备（FAQ/SOP/规则/公告）
2. 预处理：清洗 → 切分 chunk → 加元数据（来源、业务域、更新时间）
3. 建索引：embedding → 向量库
4. 检索：根据问题取 topK chunks（可加过滤：业务域/时间）
5. 生成：把 chunks 当“证据”喂给 LLM，要求**必须引用**
6. 置信度&拒答：检索分数低/证据不足 → 让模型拒答并建议人工
7. 评测：用一批固定 QA 测准确率、引用正确率

------

## 你需要搞懂的关键名词在这个项目里的位置

- **LLM**：生成回答、总结、判断是否有足够证据
- **RAG**：第 4–5 步（检索+生成）
- **spec**：输出 JSON 结构 + 引用格式 + 置信度规则
- （后续）**Agent/Tool/MCP**：不是 A 的必需品，先不引入，避免复杂化

------

## MVP（第 1 版）你只做 5 件事就能跑

### 1) 准备知识库文件

先用最简单的形式：`docs/` 目录下放一些 markdown/txt/pdf（先别上复杂格式）

- `faq.md`
- `sop_pickup.md`
- `refund_policy.md`

### 2) 切分（chunking）

规则：**按标题/段落优先**，每段 300–800 字左右，重叠 50–100 字（防断句丢上下文）。

### 3) 向量化 + 向量库

用 embedding 把每个 chunk 变成向量，存到向量库（本地用 Chroma/FAISS 都行）。

### 4) 检索 topK

输入问题，取最相关的 3–5 段证据。

### 5) 生成回答（强制引用）

提示词要求：

- 只能依据证据回答
- 每句话尽量给引用编号
- 找不到就说“不确定/建议人工”，不要编

## 推荐目录结构（工程化但不复杂）

```
kb_bot/
  docs/                # 你的知识文件
  data/
    chroma/            # 向量库本地数据
  src/
    ingest.py          # 入库：切分+embedding+写入向量库
    rag.py             # 检索+生成
    eval.py            # 跑评测集（后面加）
  README.md
```

------

## 输出 spec（非常重要，后面评测/前端都靠它）

建议你让机器人输出固定 JSON：

```json
{
  "answer": "……",
  "citations": [
    {"source": "sop_pickup.md", "chunk_id": "sop_pickup#3", "quote": "…证据片段…"}
  ],
  "confidence": "high|medium|low",
  "need_handoff": true|false,
  "handoff_reason": "…"
}
```

置信度规则（MVP 简化版）：

- **high**：top1 相似度很高 + 引用至少 2 条证据
- **medium**：有证据但只覆盖部分问题
- **low**：相似度低或证据冲突/缺失 → `need_handoff=true`

## 学习计划（2 周版，保证你能“理解并做出来”）

### 第 1–2 天：跑通 MVP

- 写 ingest：把 docs 入库
- 写 rag：问一个问题能返回“回答+引用”

### 第 3–5 天：让它变准

- 调 chunk 策略（比调模型更有用）
- 加 metadata（业务域、更新时间）
- 做 20 条小评测集（手工写）

### 第 2 周：做成可用系统

- Hybrid 检索（向量 + 关键词）
- 低分拒答策略
- 引用展示规范化
- eval 脚本每日回归
