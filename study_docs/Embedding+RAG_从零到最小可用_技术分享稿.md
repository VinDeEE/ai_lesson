# Embedding + RAG：从零到最小可用（技术分享稿）

> 适用对象：刚开始学习 AI、希望快速做出“文档问答机器人”的同学  
> 文档类型：可讲、可做、可迭代  
> 目标产出：文档 -> 向量库 -> 检索 -> 回答（含引用片段）

---

## 0. 本次分享要解决什么问题

很多人第一次做 AI 问答会遇到三个问题：

1. 模型会“瞎编”，不可靠。
2. 明明有文档，模型却答不到点上。
3. 无法解释“答案从哪来”，业务方不敢用。

Embedding + RAG 的价值是：  
把“模型会说”变成“模型基于证据说”。

---

## 1. 一句话理解 Embedding 和 RAG

- Embedding：把文本变成向量，让“语义相近”的文本在向量空间里更接近。
- RAG（Retrieval-Augmented Generation）：先检索证据，再让大模型基于证据生成答案。

可以理解为：

- Embedding 负责“把文档变成可搜索的语义坐标”。
- RAG 负责“先查资料再回答”。

---

## 2. 最小 RAG 的完整流程（MVP）

```text
原始文档
  -> 预处理（清洗、切分、加元数据）
  -> Embedding（向量化）
  -> 存入向量库
  -> 用户提问
  -> 问题向量化 + 相似度检索 topK
  -> 把检索结果作为上下文喂给 LLM
  -> 输出答案 + 引用片段 + 置信度 + 是否转人工
```

这个流程里最关键的是两个“先后顺序”：

1. 先检索，后生成。
2. 先给证据，再让模型回答。

---

## 3. 你必须掌握的 6 个核心概念

### 3.1 Chunk（分块）

为什么切分：  
整篇文档太长，检索不精确，成本也高。

推荐起步参数：

- `chunk_size`: 400~800（中文通常按字符估算）
- `chunk_overlap`: 80~150

经验：

- 太小：上下文不够，回答碎片化。
- 太大：检索噪声高，定位不准。

### 3.2 Metadata（元数据）

建议每个 chunk 至少有：

- `source_file`：来自哪个文件
- `chunk_id`：例如 `refund_policy.md#3`
- `updated_at`：更新时间（可选）
- `domain`：业务域（可选）

作用：后续可做过滤、引用展示、审计追踪。

### 3.3 Embedding Model（向量模型）

它不直接“回答问题”，而是负责“把文本变成向量”。  
同一个模型要用于：

1. 文档入库向量化
2. 用户问题向量化

否则向量空间不一致，检索效果会明显下降。

### 3.4 Vector DB（向量库）

向量库做三件事：

1. 存向量
2. 相似度检索
3. 结合 metadata 过滤

学习阶段推荐本地向量库（如 Chroma/FAISS），快速验证想法。

### 3.5 Retriever（检索器）

最小做法：相似度检索 topK（例如 K=3 或 4）。  
后续可升级：

- MMR 去重
- Hybrid（关键词 + 向量）
- Re-rank（二次排序）

### 3.6 Grounded Answer（基于证据回答）

Prompt 必须明确约束：

1. 只能根据给定证据回答。
2. 证据不足时必须拒答或转人工。
3. 必须输出引用。

## 4. “带引用片段”的统一输出规范（建议你固定下来）

```json
{
  "answer": "退款申请需要在签收后7天内提交，超时将不受理。",
  "citations": [
    {
      "source_file": "refund_policy.md",
      "chunk_id": "refund_policy.md#2",
      "quote": "用户签收后7个自然日内可申请退款..."
    }
  ],
  "confidence": "high",
  "need_handoff": false,
  "handoff_reason": ""
}
```

建议规则（MVP 版）：

- `high`：至少 2 条高相关证据，且无冲突。
- `medium`：有证据但覆盖不完整。
- `low`：证据弱、冲突、或检索命中差。
- `need_handoff=true`：`low` 且无法可靠回答时必须置为 `true`。

## 5. 最小可运行方案（学习版）

### 5.1 推荐目录

```text
kb_bot/
  docs/                 # FAQ/SOP/政策文档
  data/
    chroma/             # 本地向量库持久化目录
  src/
    ingest.py           # 入库：切分 + 向量化 + 写入向量库
    rag.py              # 检索 + 生成 + JSON输出
  tests/
    eval_qa.jsonl       # 小评测集（后续）
```

### 5.2 入库阶段做什么

1. 读取 `docs/` 下文档。
2. 文本切分为 chunk。
3. 给每个 chunk 打上 `chunk_id` 等 metadata。
4. 用 embedding 模型向量化并写入向量库。

### 5.3 问答阶段做什么

1. 将用户问题向量化。
2. 从向量库检索 topK 证据块。
3. 组装 Prompt（问题 + 证据列表 + JSON 输出要求）。
4. 调用 LLM 生成答案。
5. 校验输出 JSON，如果失败则降级兜底。

## 6. 现场分享可直接使用的 Demo 流程

### 6.1 Demo 准备（分享前）

1. 准备 3 个文档：`faq.md`、`refund_policy.md`、`delivery_sop.md`。
2. 跑一次入库脚本，确认向量库数据可用。
3. 准备 5 个问题：
- 2 个能命中
- 2 个部分命中
- 1 个知识库外问题（验证拒答）

### 6.2 Demo 演示步骤（分享时）

1. 展示流程图：文档 -> 向量库 -> 检索 -> 回答。
2. 执行一次提问命令。
3. 展示返回 JSON，重点讲 `citations`、`confidence`、`need_handoff`。
4. 提问一个知识库外问题，展示“拒答并建议人工”。

### 6.3 分享时的讲解重点

- 为什么不是“直接把问题扔给大模型”。
- 为什么 chunk 和 metadata 影响比“换大模型”更大。
- 为什么“可引用”是业务可上线的前提。

## 7. 质量评测：不做评测就无法迭代

建议先做 20 条 QA 评测集：

- 10 条可回答（答案明确）
- 5 条部分可回答（信息不完整）
- 5 条不可回答（知识库没有）

建议关注 4 个指标：

1. `retrieval_hit@k`：正确证据是否被召回
2. `answer_accuracy`：回答是否正确
3. `citation_precision`：引用是否对应答案依据
4. `handoff_accuracy`：该拒答时是否拒答

## 8. 常见坑位与修复策略

### 8.1 模型“看起来很会说”，但引用不对

原因：Prompt 约束弱，或者上下文拼接混乱。  
修复：强制 JSON 输出 + 每条结论必须有 citation。

### 8.2 检索结果“像相关但不回答问题”

原因：chunk 太大、太小，或 topK 不合理。  
修复：先调 chunk，再调 topK，再考虑换模型。

### 8.3 文档更新后回答还是旧内容

原因：未重建索引或没有版本控制。  
修复：文档更新触发增量入库，记录 `updated_at`。

### 8.4 知识库外问题仍被强行回答

原因：没有阈值和拒答规则。  
修复：加相似度阈值、证据条数阈值，低分强制 handoff。

## 9. 从 MVP 到可用系统的演进路线

第 1 阶段（本周）：跑通

1. 本地文档入库
2. topK 检索
3. JSON 回答 + 引用

第 2 阶段（下周）：变准

1. 优化 chunk 和 metadata
2. 加评测集 + 每日回归
3. 加拒答阈值

第 3 阶段（再下一步）：可上线

1. Hybrid 检索
2. 权限过滤（按部门/角色）
3. 监控与日志审计（问题、证据、答案）

## 11. 可直接复制的 Prompt 模板（学习版）

```text
你是文档问答助手。你只能根据“证据片段”回答，不允许编造。

规则：
1) 如果证据不足，请输出 need_handoff=true，并解释原因。
2) 回答要简洁、可执行。
3) 每个关键结论都要在 citations 中给出证据。
4) 只输出 JSON，不要输出任何额外文字。

问题：
{question}

证据：
{context}

输出格式：
{
  "answer": "...",
  "citations": [{"source_file":"...","chunk_id":"...","quote":"..."}],
  "confidence": "high|medium|low",
  "need_handoff": true|false,
  "handoff_reason": "..."
}
```

## 12. 下一轮细化清单（持续迭代用）

每次迭代建议只改 1~2 项并记录结果：

1. chunk 参数（size/overlap）
2. 检索参数（topK/阈值/MMR）
3. Prompt 约束（引用与拒答）
4. 文档质量（结构化、去噪、更新频率）
5. 评测集规模（20 -> 50 -> 100）

```text
- ...

- ...
```

---

## 13. 结语（分享时可直接说）

RAG 的核心不是“让模型更聪明”，而是“让回答可追溯、可验证、可迭代”。  
先做最小闭环，再用评测驱动优化，你就能把一个 Demo 稳定推进到可用系统。
