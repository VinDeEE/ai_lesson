# AI 分享选题收敛与对话记录

更新时间：2026-03-24

## 背景

目标是给组内做一次 AI 相关分享。

已知约束：

- 分享对象主要是非 AI 领域同学
- 已经有人分享过 `Transformer` 和“模型微调”
- 希望听众有获得感，降低大家的学习成本
- 不要求题目高大上，也不要求一定讲满 1 小时
- 更希望针对一个具体内容讲清楚

## 对话过程记录

### 第一轮需求

初始诉求是：

- 作为十年 Java 程序员，想学习 AI 相关知识
- 希望了解现在 AI 领域需要学习哪些内容
- 需要选一个内容做组内分享
- 分享时间大约 1 小时

基于这个诉求，最开始给出的建议偏“学习路线 + 热门方向”，包括：

- LLM 应用基础
- RAG
- Agent 工程化
- MCP
- Evals
- 多模态
- 安全治理

并给出了一个偏大的候选题目：

**AI Agent 工程化：从 Tool Calling 到 MCP，Java 如何落地企业级智能应用**

### 第二轮反馈

随后提出了一个很关键的约束：

**这个主题可能太宽泛、太空，分享需要让组员有获得感，减少学习成本，不需要高大上。**

这一步实际上把选题标准从“前沿、完整”切换成了“具体、易懂、可带走”。

根据这个反馈，对题目重新收敛，判断如下：

1. `Agent + MCP + Workflow` 适合作为认知扩展，但对一次内部分享来说偏大。
2. 如果内容太大，听众容易只记住名词，难形成可复用判断。
3. 组内分享更适合讲一个大家见过、但说不清原理的问题。

### 第三轮收敛

进一步明确：

- 不一定非要 1 小时
- 30 到 45 分钟也可以
- 重点是把某一个内容讲清楚
- 适合选“非 AI 同学比较感兴趣，或者平时不好理解的点”

到这一步，最终的选题策略已经比较明确：

**不讲大而全的 AI 地图，改讲一个大家日常会碰到、且容易误解的问题。**

## 最终推荐题目

### 题目

**为什么 RAG 经常答非所问：从文档切分、检索到重排，讲清 AI 知识库到底是怎么工作的**

## 为什么这个题目更适合当前分享

这个题目比 Agent、MCP 更适合当前这次分享，原因是：

1. 大家对“AI 接了知识库为什么还会答错”天然有兴趣。
2. 问题足够具体，不容易讲空。
3. 不需要太多模型背景，后端、测试、产品都能听懂。
4. 听完之后能直接提升判断力，知道知识库效果不好时该怀疑哪里。
5. 这个题目能明显降低大家的学习成本，因为它把一个常见现象拆成可理解的工程环节。

## 这个分享希望解决的核心问题

建议围绕下面几个问题展开：

- 文档明明存在，为什么模型就是答不出来？
- 为什么切分方式会直接影响回答质量？
- 为什么“相似度高”不等于“检索正确”？
- 为什么模型会把检索到的内容理解歪？
- 为什么有时候不是模型不行，而是检索链路有问题？

这几个问题都很具体，也很容易引发讨论。

## 适合的分享时长

这次不建议硬凑 1 小时。

更合适的长度是：

- 30 分钟：快速讲清原理和常见坑
- 40 分钟：原理 + 常见坑 + 简单改进方案
- 45 分钟：原理 + 常见坑 + 改进方案 + 简短问答

如果内容控制得好，**40 分钟左右通常是最合适的**。

## 推荐分享结构

### 1. 先讲现象，5 分钟

从一个大家熟悉的问题切入：

“为什么 AI 明明接了知识库，还是答非所问，甚至一本正经胡说八道？”

这一步不要急着讲术语，先让听众代入问题。

### 2. 讲最小原理，10 分钟

只讲 RAG 最核心的链路：

1. 文档切分
2. 向量化
3. 检索召回
4. 重排
5. 拼接上下文
6. 生成答案

让大家先知道知识库问答不是“把 PDF 丢进去就结束了”，而是一个流水线。

### 3. 讲最常见的坑，15 分钟

建议重点讲 5 个：

1. Chunk 切得太大，召回不准
2. Chunk 切得太碎，上下文断裂
3. 只做向量检索，不做重排
4. 上下文塞太多，噪音淹没有效信息
5. 回答不附来源，用户无法判断可信度

这部分是全场最有获得感的内容。

### 4. 讲工程改法，10 分钟

不必讲太深，讲实用原则即可：

- 按语义切分，不要机械按字数切
- 给文档加 metadata
- 控制 topK，不是越多越好
- 在关键场景里加重排
- 回答尽量附来源
- 优先解决检索质量，再讨论模型问题

### 5. 总结，3 到 5 分钟

最后落一句最重要的话：

**很多知识库效果差，不是模型不够强，而是检索链路没有设计好。**

## 这次分享能给组员带来的获得感

如果按这个题目讲，听众至少能带走下面这些东西：

1. 知道 RAG 不是“接个向量库”那么简单。
2. 知道知识库答错时，排查应该从哪里开始。
3. 知道切分、召回、重排、上下文拼装分别在干什么。
4. 知道为什么 AI 项目里很多问题其实是工程问题，不全是模型问题。

这类获得感比知道更多前沿名词更实在。

## 为什么不推荐这次继续讲大题目

下面这些题目本身没有问题，但不适合当前目标：

- Agent
- MCP
- 多 Agent
- AI 学习路线全景图

原因不是它们不重要，而是：

1. 概念面太广
2. 需要先铺很多背景
3. 听众短时间内不容易形成稳定认知
4. 对“减少学习成本”帮助不如一个具体问题大

## 如果还想要一个备选题目

如果不讲 RAG，也可以讲这个：

**Function Calling 到底解决了什么：为什么 AI 要从聊天变成调接口**

这个题目也很适合 Java 团队，因为它天然贴近接口设计、结构化输出和系统集成。

但如果只选一个，当前仍然更推荐：

**为什么 RAG 经常答非所问**

## 最终结论

这次分享不需要追求“讲得很大”或“讲满 1 小时”，更适合：

**用 30 到 45 分钟，把一个大家最容易遇到、但又说不清的问题讲清楚。**

当前最推荐的正式题目是：

**《为什么 RAG 经常答非所问：从文档切分、检索到重排，讲清 AI 知识库到底是怎么工作的》**

这个题目更具体、更容易让非 AI 同学听懂，也更能让大家在分享结束后真正带走东西。
