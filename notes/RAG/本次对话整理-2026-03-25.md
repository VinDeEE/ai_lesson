## 2. 主要讨论内容

### 2.1 运行 `src/ingest.py` 时的接口问题

先排查了入库阶段的 Gemini 接口报错，先后确认了两类问题：

- `HTTP 429`：Embedding 免费额度或分钟级限流触发
- `HTTP 400 API_KEY_INVALID`：旧 API key 失效

后续验证发现：

- 当前 `.env` 中的新 key 可以正常访问 Gemini
- `models` 请求可用
- `embedContent` 请求可用

结论：

- 新 key 本身是有效的
- 之前的 key 无效问题是旧进程或旧 key 导致的

---

### 2.2 对 `rag_core.py` 的稳定性修正

为了让 demo 更稳，做了以下改动：

- 为 Gemini 请求增加了 429 自动等待重试
- 避免在限流时错误地退回成更碎的单条 embedding 请求
- 修复了问答链路中提示词和 JSON 输出解析问题

这些修改的目的是：

- 让入库阶段更稳
- 让问答阶段输出更可控
- 减少分享时因为接口波动导致的异常

---

### 2.3 `ask.py` 调用链的梳理

对当前问答链路做了明确拆解：

1. `ask.py` 负责接收命令行参数
2. `answer_question()` 负责整体编排
3. `load_index()` 读取本地索引
4. `embed_texts()` 把用户问题向量化
5. 本地对全部 chunk 做余弦相似度排序
6. `call_chat_model()` 基于召回证据调用生成模型
7. 输出结构化 JSON

核心认知：

- 一次问答至少包含两次模型调用
- 检索层和生成层是两段不同职责的流程

---

### 2.4 `evaluate.py` 的定位

解释了 `src/evaluate.py` 的作用：

- 不是单题问答脚本
- 而是一个批量回归验证工具

它会：

- 读取 `eval/qa_samples.jsonl`
- 逐条调用 `answer_question()`
- 根据关键词命中和 `need_handoff` 命中做简单评分

结论：

- 它适合做 smoke test
- 不等于完整评测平台

---

### 2.5 `notes` 目录对向量化的影响

讨论并确认了一个重要事实：

- 之前 `notes` 放在 `docs/` 下
- `ingest.py` 默认扫描整个 `docs/`
- 所以学习记录也会被一起切块并入库

这会造成：

- chunk 数变大
- 检索噪音变多
- 工单问答可能召回到学习资料

因此建议：

- `notes` 不应继续放在 `docs/` 下参与入库
- 分享时要明确说明当前历史索引曾受到这一点影响

### 2.6 为什么 `chunk_count = 243`

解释了 `243` 的来源：

- 当时共读取了 `36` 个 `.md/.txt` 文件
- 每个文件按 `chunk_size=600`、`chunk_overlap=120` 做切块
- 所有文件切出来的 chunk 汇总后总数为 `243`

这个数字可以在两个地方看到：

- `data/chroma/kb.json` 中的 `chunk_count`
- 重新执行 `python src/ingest.py` 时的 `[OK] chunks: ...` 输出

### 2.7 增加 `--vector_only` 模式

为了做技术分享中的效果对比，新增了只用向量、不调用生成模型的模式。

现在可以用两条命令对比：

```powershell
python src/ask.py "工单时效有问题"
python src/ask.py "工单时效有问题" --vector_only
```

两种模式的差异：

- 默认模式：检索后再调用生成模型输出摘要化答案
- `--vector_only`：只返回 top-k 原始召回片段

这个改动的核心价值是：

- 能把检索层和生成层拆开讲
- 能非常直观地演示“大模型到底增加了什么”

### 2.10 澄清当前 demo 到底做了什么
围绕“这个 demo 具体做了哪几件事、顺序是什么、依赖什么、什么时候调外部服务”又做了一次专门梳理，核心结论是：

- 当前 demo 实际只做 3 件事：离线入库、在线问答、批量评测
- 离线入库入口是 `src/ingest.py` / `src/week3_ingest_runner.py`
- 在线问答入口是 `src/ask.py`，核心在 `src/rag_core.py`
- 批量评测入口是 `src/evaluate.py`

最关键的依赖关系是：

```text
docs/*
  -> ingest.py
  -> data/chroma/kb.json
  -> ask.py / evaluate.py
```

也就是说：

- 先有源文档
- 再执行入库
- 生成本地索引 `kb.json`
- 之后问答和评测都只依赖本地索引，不直接读 `docs/`

在线问答的实际顺序被明确为：

```text
用户问题
-> ask.py
-> answer_question()
-> load_index(kb.json)
-> embed_texts([question])
-> 本地遍历全部 chunk 做 cosine_similarity
-> 取 top_k
-> 分支 A: --vector_only，直接返回召回片段
-> 分支 B: build_context + call_chat_model()
-> 解析 JSON
-> 输出 answer / citations / confidence / need_handoff
```

这次也把“外部服务”和“本地逻辑”的边界讲清楚了：

- 外部服务只有 Gemini Embedding API 和 Gemini Generate API
- 入库时只调用 embedding
- 正常问答时先调用 embedding，再调用生成模型
- `--vector_only` 模式只调用 embedding，不调用生成模型
- `evaluate.py` 本质上是循环执行问答流程

本地逻辑则包括：

- 读取 `.env`
- 扫描文档
- 切块
- 保存和加载 `kb.json`
- 本地余弦相似度检索
- prompt/context 组装
- JSON 解析与兜底
- citations / confidence / need_handoff 构造
- 429 重试和简单评分

这部分澄清非常关键，因为它把当前 demo 的真实边界说清楚了：

- 当前不是向量数据库方案
- 当前不是 ANN 检索
- 当前是“本地 JSON 索引 + Gemini API”的最小闭环 RAG demo

## 3. 这次对话沉淀下来的关键结论

### 3.1 关于当前 RAG demo

- 当前 demo 已经具备最小闭环
- 但当前检索仍是本地全量扫描，不是生产级方案
- 当前索引曾受学习笔记混入影响

### 3.2 关于“为什么要有大模型”

核心结论：

**向量检索解决的是“找到什么”，大模型生成解决的是“怎么回答”。**

### 3.3 关于分享策略

最值得讲的不是“功能有多少”，而是：

- 架构怎么搭
- 数据怎么流
- 工单怎么被处理
- 为什么需要检索和生成协同
- 当前方案的边界在哪里
