# RAG 技术团队分享文档

## 标题

**从 0 到 1 理解 RAG：原理、架构、落地与常见问题**

---

## 1. 分享背景

### 1.1 为什么要讲 RAG

在大模型落地过程中，大家很快会遇到几个典型问题：

- 模型回答看起来很合理，但内容是错的
- 模型不知道公司的私有知识
- 业务知识更新很快，不能靠重新训练模型来解决
- 用户问法很多样，单纯关键词检索效果不稳定
- 上线后很难评估“到底有没有真的变好”

RAG（Retrieval-Augmented Generation，检索增强生成）就是解决这类问题的主流方案。

### 1.2 这次分享希望解决什么问题

通过这次分享，希望团队能统一以下认知：

1. RAG 不是“接一个向量库”那么简单
2. 一个可用的 RAG 系统，本质上是“检索系统 + 上下文构造 + 生成系统 + 评估体系”的组合
3. RAG 的上限，很多时候不在模型，而在数据、切片、召回、重排和评估
4. 做 RAG 要优先关注业务目标，而不是盲目堆技术名词

---

## 2. 什么是 RAG

### 2.1 一句话定义

RAG 是一种让大模型在回答问题前，先去外部知识源中检索相关内容，再基于检索结果进行生成的技术方案。

### 2.2 为什么需要 RAG

大模型本身存在几个天然限制：

- **参数知识静态**：训练完成后，知识不是实时更新的
- **缺少私域知识**：不知道企业内部文档、知识库、FAQ、工单经验
- **容易幻觉**：在不知道时也可能生成“像真的一样”的答案
- **可解释性弱**：不容易说明答案来自哪里

RAG 的核心思路是：

> 不要求模型“记住所有知识”，而是让模型“在回答前先查资料”。

### 2.3 和微调的区别

| 方案 | 解决的问题 | 优点 | 缺点 | 适合场景 |
|---|---|---|---|---|
| Prompt Engineering | 改善任务指令 | 成本低、迭代快 | 不能补充新知识 | 通用任务优化 |
| Fine-tuning | 学习固定风格/模式 | 稳定、可针对任务优化 | 成本高、知识更新慢 | 格式化输出、分类、特定风格 |
| RAG | 引入外部知识 | 更新快、可追溯、适合私域知识 | 架构更复杂，依赖检索质量 | 知识问答、客服、Copilot、企业搜索 |

### 2.4 一个直观例子

用户问：

> “退款审核通过后多久可以到账？”

如果只靠大模型：

- 可能回答一个常识性但不准确的时间
- 也可能忽略平台/业务线差异

如果使用 RAG：

- 先在退款规则、FAQ、流程文档中检索
- 找到“不同支付方式到账时效”的规则
- 再基于这些规则生成答案，并附上来源

这样回答会更稳、更可控。

## 3. RAG 的核心流程

一个标准 RAG 流程通常分成两条链路：

- **离线链路**：知识加工
- **在线链路**：问题回答

### 3.1 离线链路：把知识变成可检索的数据

典型流程：

1. 数据接入
2. 文档解析
3. 清洗去噪
4. 切片（Chunking）
5. 向量化（Embedding）
6. 建索引
7. 元数据存储

### 3.2 在线链路：收到问题后怎么回答

典型流程：

1. 用户输入问题
2. Query 改写/补全
3. 多路召回（关键词、向量、标签过滤等）
4. 重排（Rerank）
5. 上下文拼装
6. 调用大模型生成答案
7. 引用来源 / 置信度控制 / 安全兜底

### 3.3 用一句话概括

> 离线解决“知识怎么存”，在线解决“知识怎么找、怎么喂给模型、怎么安全地答”。

## 4. RAG 的系统架构拆解

### 4.1 典型架构图（文字版）

```text
               ┌──────────────────────┐
               │   文档/FAQ/工单/知识库 │
               └─────────┬────────────┘
                         │
                   文档解析与清洗
                         │
                    Chunk 切片处理
                         │
                    Embedding 向量化
                         │
          ┌──────────────┴──────────────┐
          │                             │
      向量索引库                    元数据/倒排索引
          │                             │
          └──────────────┬──────────────┘
                         │
用户问题 → Query 改写 → 多路召回 → Rerank → Context 构造 → LLM 生成 → 返回答案
```

### 4.2 关键模块说明

#### A. 文档接入层

负责接入不同知识源：

- FAQ
- 产品文档
- 帮助中心
- 内部 Wiki
- 工单历史
- 聊天记录
- 标准话术
- 业务规则配置

挑战：

- 数据格式不统一
- 文档质量参差不齐
- 很多内容是“给人看的”，不是“给机器检索的”

#### B. 解析与清洗层

主要任务：

- 去掉无意义内容（页眉页脚、版权、导航项）
- 结构化标题、段落、表格
- 提取知识属性（产品线、业务类型、生效时间、适用范围）
- 去重与版本控制

#### C. Chunk 切片层

这是 RAG 成败的关键环节之一。

切片不是简单按字数截断，而是要平衡：

- 语义完整性
- 检索召回率
- 上下文长度
- 后续生成可用性

常见切法：

- 固定长度切片
- 滑动窗口切片
- 按标题层级切片
- 按语义段落切片
- 表格/问答对专项切片

#### D. 检索层

常见检索方式：

- **关键词检索（BM25）**：适合精准术语、错误码、产品名
- **向量检索**：适合语义相近表达
- **混合检索（Hybrid Search）**：线上最常见
- **过滤检索**：基于 metadata 限定业务线、语言、时间等

#### E. 重排层

召回不等于最终可用。

重排的作用是：

- 从一批候选片段中挑出“最适合当前问题”的片段
- 解决向量召回“语义相关但不够直接”的问题
- 提升 topN 结果质量，减少无关上下文进入模型

#### F. 上下文构造层

不是把召回结果无脑拼接给模型，而是要考虑：

- 排序顺序
- 去重
- 冗余压缩
- token 预算
- 是否保留标题、来源、时间、生效范围
- 是否要做证据摘要

#### G. 生成与控制层

模型生成时要控制：

- 是否必须基于已检索内容回答
- 检索不到时是否允许回答
- 是否输出引用
- 是否给出“不确定/建议转人工”
- 是否进行敏感内容过滤

## 5. RAG 的关键技术点

## 5.1 Chunk 怎么切

### 一个错误认知

很多人刚做 RAG 时，会把文档按 500 字或 1000 字硬切。这样做虽然简单，但效果常常不好。

### 为什么切片重要

切得太大：

- 检索命中不够精准
- 无关信息太多
- 模型容易被噪声干扰

切得太小：

- 语义不完整
- 召回到的内容无法独立回答问题
- 上下文需要拼很多片段，复杂度升高

### 实践建议

- FAQ 类：优先按问答对切
- 说明文档类：按标题层级 + 段落切
- 流程类：保留步骤完整性
- 表格类：不要简单转纯文本，要保留字段关系
- 长文档：适当做 overlap，避免跨段断义

### 常用经验

- chunk size 不是越大越好，也不是越小越好
- 更重要的是“一个 chunk 是否能表达一个相对完整的知识单元”

---

## 5.2 Query 改写为什么很重要

用户提问往往不标准：

- 口语化
- 指代不清
- 缺少业务关键词
- 带上下文省略

例如：

> “这个能退吗？”

直接去检索几乎没有效果。

但如果结合会话上下文改写成：

> “生鲜团购订单发货后是否支持退款？”

召回质量会明显提升。

### 常见改写方式

- 会话补全
- 术语归一
- 同义词扩展
- 问题标准化
- 多查询扩展（Multi-query）

### 风险

改写本身也可能引入偏差，因此要注意：

- 改写后尽量保留原意
- 高风险场景中可保留原 query + 改写 query 双路召回

---

## 5.3 为什么要混合检索

只做向量检索的问题：

- 业务词、产品名、错误码、规则编号等未必稳
- 对高精度场景不够可靠

只做关键词检索的问题：

- 用户表达稍微变一下就召回不到
- 语义理解弱

所以实际中最常见的是：

> BM25 + 向量召回 + metadata filter + rerank

这是兼顾召回率和精度的主流组合。

---

## 5.4 Rerank 为什么常常比换模型更有价值

很多 RAG 问题，不是模型不会答，而是喂给模型的上下文不对。

如果 top10 召回里有正确答案，但排位靠后，模型实际上可能根本没看到，或者看到了也被噪声干扰。

这时增加一个 rerank，收益往往比：

- 换更大的生成模型
- 拉长 prompt
- 一味扩大 topK

更直接、更稳定。

### 可以这样理解

- **召回**：尽量别漏
- **重排**：尽量把最相关的排前面
- **生成**：基于优质证据组织答案

---

## 5.5 Prompt 在 RAG 中真正该做什么

很多团队对 Prompt 有误解，认为 Prompt 能解决所有问题。

在 RAG 中，Prompt 更适合做的是：

- 限定回答边界
- 规定输出格式
- 明确引用要求
- 控制拒答策略
- 约束语言风格

而不是弥补：

- 错误召回
- 脏数据
- 切片不合理
- 文档版本混乱

一个常见原则：

> Prompt 决定回答方式，检索决定回答依据。

## 6. RAG 常见问题与坑

## 6.1 检索到了，但还是答不对

常见原因：

- 召回片段太多，噪声干扰
- 正确片段排位太后
- chunk 不完整
- prompt 没有限制模型必须依据材料回答
- 多份文档内容冲突，模型自行“综合”后说错

### 解决思路

- 优化 rerank
- 压缩上下文
- 引入版本/生效时间过滤
- 强制答案引用证据
- 高风险问题启用保守策略

---

## 6.2 检索不到

常见原因：

- 文档本身没有这条知识
- 文档有，但切片方式导致无法命中
- query 表达和知识表达差异太大
- embedding 模型不适配业务语料
- 元数据错误，过滤条件把结果挡掉了

### 解决思路

- 优化 query 改写
- 做混合召回
- 调整 chunk 策略
- 增加业务术语词典
- 完善知识建设流程

---

## 6.3 引入了很多知识，效果还是不稳定

这通常说明问题不在“知识量不够”，而在：

- 知识质量差
- 版本不统一
- 粒度不合适
- 召回链路不稳定
- 没有评估闭环

RAG 不是“把所有文档丢进向量库”就结束了。

---

## 6.4 多轮对话下效果下降

原因通常包括：

- 当前问题依赖上文指代
- 会话历史过长，噪声增加
- 历史问题影响当前 query 意图识别

### 常见处理方式

- 对当前轮做 query 重写
- 有选择地压缩会话历史
- 区分“会话记忆”和“知识检索”两类上下文

---

## 6.5 文档版本冲突

这是企业场景里非常常见的问题。

比如：

- 老规则还在库里
- 新旧流程同时存在
- 不同业务线规则相似但不完全一致

如果不做版本和范围控制，模型很容易答错。

### 建议

索引中要保留清晰 metadata：

- 生效时间
- 失效时间
- 业务线
- 区域
- 产品类型
- 文档版本号
- 权威等级

## 7. RAG 在业务场景中的典型应用

## 7.1 企业知识问答

适合场景：

- 内部知识库问答
- 制度流程问答
- 产品手册查询
- 运维排障知识查询

价值：

- 降低检索成本
- 提升知识获取效率
- 降低对专家的依赖

## 7.2 智能客服

适合场景：

- 售前售后 FAQ
- 规则解释
- 流程指引
- 订单/退款/配送等政策问答

价值：

- 降低人工服务压力
- 缩短响应时间
- 提升回答一致性

## 7.3 坐席辅助

适合场景：

- 实时推荐知识
- 自动生成回复建议
- 工单处理辅助
- 通话/会话中智能提示

价值：

- 提升人工处理效率
- 缩短培训周期
- 减少经验依赖

## 7.4 工单与历史案例检索

适合场景：

- 相似工单推荐
- 故障案例查询
- 历史处理经验复用

价值：

- 提升问题闭环效率
- 让“经验”可复用

## 8. 做一个可落地 RAG，需要关注哪些指标

RAG 系统不能只看“模型回答像不像”，要拆成几个层次评估。

## 8.1 检索层指标

常见指标：

- Recall@K
- Precision@K
- MRR
- NDCG
- 命中率
- 首条命中率

这些指标回答的是：

> 正确证据有没有被召回？排得够不够靠前？

## 8.2 生成层指标

常见评估维度：

- 是否回答正确
- 是否引用了有效证据
- 是否有幻觉
- 是否完整
- 是否符合业务规则
- 是否可直接给用户看

## 8.3 业务层指标

真正上线后更应该看：

- 问题解决率
- 转人工率
- 一次解决率
- 用户满意度
- 平均处理时长
- 坐席采纳率
- 错答率 / 高风险错答率

### 一个重要原则

> RAG 的好坏，最终要落在业务收益上，而不是只看离线 benchmark。

## 9. 一个比较实用的 RAG 落地方法论

这里给一个适合技术团队推进的路线。

### 第一步：先明确业务目标

不要一上来问“用哪个向量库”。

先回答这些问题：

- 我们要解决什么业务问题？
- 是查不到，还是答不准，还是答得慢？
- 错答的代价高不高？
- 用户问的是标准问题还是开放问题？

### 第二步：明确知识边界

搞清楚：

- 知识源有哪些
- 哪些是权威源
- 哪些是低质量源
- 是否有版本、时效、适用范围

### 第三步：先做最小闭环

建议从一个最小可控场景切入：

- 问题范围有限
- 知识相对稳定
- 能快速构建评测集
- 业务收益清晰

比如：

- 退款规则问答
- 售后政策问答
- 内部 SOP 查询

### 第四步：优先打磨检索，不要急着换模型

很多项目初期，80% 的问题都出在：

- 数据
- 切片
- 召回
- 重排
- 上下文构造

而不在模型本身。

### 第五步：建立评估闭环

至少要有：

- 标准问题集
- 标准答案或标准证据
- 离线评估
- 线上抽样评估
- bad case 归因
- 持续迭代机制

---

## 10. 一个典型的技术方案示例

以下是一个工程上比较常见的方案组合，仅供参考：

### 离线部分

- 文档解析：自研解析 / 第三方解析组件
- 文本切片：按标题层级 + 段落 + overlap
- 向量模型：通用 embedding 或领域优化 embedding
- 索引：向量库 + 倒排索引
- metadata：业务线、文档类型、版本、生效时间、标签

### 在线部分

- query 改写：基于规则或小模型
- 召回：BM25 + 向量混合召回
- 重排：cross-encoder reranker 或大模型判分
- prompt：严格限制基于证据回答
- 输出：答案 + 来源 + 置信度/兜底策略

### 监控部分

- 检索耗时
- LLM 耗时
- topK 命中分布
- 无结果率
- 高风险问题占比
- 错答样本归因

## 11. RAG 不是银弹

虽然 RAG 很重要，但它不适合解决所有问题。

### 不太适合单靠 RAG 的场景

- 复杂推理链特别长的问题
- 需要强流程执行的任务
- 实时操作型任务（不仅仅是问答）
- 高精度结构化决策场景
- 强个性化、多系统状态联动场景

这类场景通常需要结合：

- Tool Calling
- Workflow / Agent
- 规则引擎
- 业务系统 API
- 微调或任务专项模型

### 一个更准确的定位

> RAG 擅长解决“知识获取与知识利用”问题，不擅长单独解决“复杂决策与系统执行”问题。

---

## 12. 适合团队达成的几个共识

最后可以给团队沉淀几个核心共识：

### 共识一：RAG 的核心是“检索质量”

不是只看模型，也不是只看向量库。

### 共识二：知识工程决定上限

数据质量、结构化程度、版本管理、切片方式，决定了系统能否稳定。

### 共识三：评估体系必须先行

没有评估，就无法知道优化是否有效。

### 共识四：业务场景要分层治理

不同问题：

- 能自动答的自动答
- 高风险问题保守答
- 不确定问题要明确兜底

### 共识五：RAG 是一套系统工程

它涉及：

- 数据治理
- 检索工程
- Prompt 设计
- 大模型调用
- 评估与监控
- 业务规则控制

不是单点技术。

## 13. 分享总结

如果用几句话总结今天的内容：

1. RAG 的本质是让大模型“先查资料，再回答”
2. 一个好用的 RAG，关键不在模型多大，而在检索链路是否扎实
3. RAG 落地的重点是：数据、切片、召回、重排、上下文构造、评估
4. 在企业和客服场景中，RAG 非常适合做知识问答、坐席辅助和案例检索
5. 真正要把效果做稳，必须把它当作一个系统工程来建设

## 14. 可直接用于现场分享的结束语

RAG 之所以重要，不是因为它是一个热门名词，而是因为它提供了一个现实可行的路径：

**不要求模型记住一切，而是让模型具备基于知识进行可靠回答的能力。**

对于技术团队来说，RAG 的价值不只是在“接入大模型”，更在于借这个过程倒逼我们重新思考：

- 知识是否可治理
- 数据是否可复用
- 经验是否可沉淀
- 业务规则是否可被系统化表达

这也是它真正的工程价值。

## 16. 一页版极简总结（适合做最后一页 PPT）

### RAG = 检索 + 生成

先从知识库中找到相关内容，再让大模型基于内容作答。

### RAG 解决的问题

- 大模型不知道私域知识
- 知识更新慢
- 容易幻觉
- 难以追溯答案来源

### RAG 的关键模块

- 文档解析
- Chunk 切片
- 混合召回
- Rerank 重排
- Context 构造
- 生成控制
- 评估闭环

### RAG 落地重点

- 数据质量
- 检索质量
- 版本治理
- 高风险兜底
- 业务效果验证

### 一句话结论

**RAG 不是一个模型技巧，而是一套面向知识利用的系统工程。**
