## 第 3 页：RAG 是什么

### 标题

RAG 的本质：先找证据，再组织答案

### 页面文案

- RAG = Retrieval-Augmented Generation
- 中文通常叫“检索增强生成”
- 核心思想：
- 先从外部知识源中检索相关证据
- 再基于证据让大模型生成答案

一句话区分：

- 纯大模型：靠参数记忆回答
- RAG：靠“外部证据 + 模型生成”回答

### 讲解提示

RAG 不是让模型凭空更聪明，而是让模型回答时带着资料、带着依据、带着可追溯的上下文去回答。

## 第 4 页：为什么企业需要 RAG

### 标题

企业为什么不能只靠纯大模型

### 页面文案

- 企业知识很多是私域知识
  - FAQ、SOP、制度、工单规则、产品文档
- 企业知识变化快
  - 模型参数不会自动更新
- 业务场景不能接受明显幻觉
  - “看起来合理但实际上错误”的答案风险很高
- 企业对权限、安全、合规要求更高
- 企业回答往往需要可追溯、可审计

### 讲解提示

企业真正需要的不是一个“会聊天”的模型，而是一个能使用企业知识、能给出依据、能被治理的知识系统。

## 第 5 页：RAG 的价值与边界

### 标题

RAG 擅长什么，不擅长什么

### 页面文案

适合 RAG 的场景：

- FAQ、知识库问答
- 企业制度、流程、SOP 查询
- 客服辅助、工单辅助
- 产品、研发、运维知识问答
- 需要给出出处和证据的回答场景

不一定适合 RAG 的场景：

- 纯创作类内容
- 完全不依赖外部知识的开放聊天
- 极短且固定模板化的问答
- 对时延极端敏感且知识变化很少的场景

### 讲解提示

这页的重点是告诉大家，RAG 不是万能解法，它更适合“知识明确、需要证据、需要可控”的场景。

## 第 6 页：RAG 的发展过程

### 标题

RAG 是怎么演进到今天的

### 页面文案

- Retrieve-and-Read
- 模型型 RAG
- 应用框架型 RAG
- Pipeline / Workflow 型 RAG

这条演进线说明：

### 讲解提示

今天市面上的 RAG 框架差异很大，本质原因就是它们站在这条演进线的不同阶段上。

## 第 7 页：企业级 RAG 总体架构

### 标题

一个企业级 RAG 系统至少包含 4 层能力

### 页面文案

- 数据层
  - 数据接入、清洗、切块、向量化、索引
- 检索层
  - 召回、过滤、重排、融合
- 生成与编排层
  - Prompt 编排、结构化输出、引用控制
- 治理层
  - 评测、权限、监控、审计、成本治理

一句话总结：

企业级 RAG 不只是“向量检索 + 大模型”，而是一套完整系统。

### 讲解提示

这一页最重要的是把听众从“RAG 只是一个链路”提升到“RAG 是一套系统”的认知。

## 第 8 页：企业级 RAG 的完整流程

### 标题

离线构建知识，在线回答问题

### 页面文案

离线阶段：

- 接入业务文档
- 清洗和结构化
- 切块 Chunking
- 生成 Embedding
- 写入索引 / 向量库

在线阶段：

- 用户提问
- Query Embedding
- Retrieve / Rerank
- Prompt 组装
- LLM 生成
- 输出答案、引用和控制字段

### 讲解提示

很多人第一次接触 RAG 时，容易把它理解成一次实时调用。实际上 RAG 是典型的“离线准备 + 在线使用”双链路系统。

## 第 11 页：当前 demo 的完整问答链路

### 标题

一次问答是怎么跑完的

### 页面文案

在线问答流程：

- 用户问题进入 `ask.py`
- `answer_question()` 读取 `kb.json`
- 调用 embedding 把问题转成向量
- 本地遍历全部 chunk 做相似度排序
- 取 top-k 证据
- 组装 context / prompt
- 调用 Gemini Generate
- 输出 `answer / citations / confidence / need_handoff`

本地负责：

- 加载索引
- 相似度计算
- prompt 组装
- JSON 解析

外部服务负责：

- Gemini Embedding
- Gemini Generate

### 讲解提示

这一页的关键是把“本地逻辑”和“模型服务”分清楚。RAG 不是所有事情都交给模型做，很多关键控制逻辑其实在本地。

## 第 12 页：有模型 vs 无模型

### 标题

为什么要保留 vector_only 模式

### 页面文案

两种模式：

- 默认模式：检索 + 生成
- `--vector_only`：只检索，不生成

它们的差异不在“检索结果一定不同”，而在“最终输出形式不同”：

- 无模型模式：返回原始召回片段
- 有模型模式：返回整理后的结构化答案

一句话结论：

**向量检索解决的是“找到什么”，大模型生成解决的是“怎么回答”。**

### 讲解提示

这是最适合现场演示的一页，也最容易让听众真正理解大模型在 RAG 里增加了什么价值。

## 第 13 页：当前 demo 的边界

### 标题

这个 demo 已经能讲清原理，但还不是生产级系统

### 页面文案

当前方案的边界：

- 还不是真正的向量数据库方案
- 检索还是全量扫描，不是 ANN
- 没有 BM25 / sparse retrieval
- 没有 hybrid retrieval
- 没有 rerank
- 没有权限治理
- 评测还比较轻

这页的核心意思：

- demo 的价值在于讲清原理
- 不是在于它已经具备完整生产能力

### 讲解提示

把这页讲清楚，听众会更容易接受后面“为什么还要继续工程化”。

## 第 14 页：从 demo 到工程化

### 标题

从能跑到能落地，还要补哪些能力

### 页面文案

检索层增强：

- ANN
- BM25 / sparse retrieval
- hybrid retrieval
- rerank
- query rewrite

数据层增强：

- 增量入库
- 去重
- metadata
- 文档版本与更新时间管理

生成层增强：

- 更稳定的 schema
- 输出校验
- handoff 策略优化

评测与平台层增强：

- Recall@K、MRR、Faithfulness、Citation Precision
- 权限控制、多知识库、缓存、监控、成本治理

### 讲解提示

这一页不要讲成“功能罗列”，而要讲成“路线图”。从 demo 到生产级，最关键的是把检索、数据、评测和治理做成体系。

## 第 15 页：总结与 Q&A

### 标题

一页总结

### 页面文案

1. RAG 的本质是“先找证据，再组织答案”
2. 企业需要 RAG，是因为纯大模型缺少私域知识、时效性、可控性和可追溯性
3. 一个完整的 RAG 系统包含数据层、检索层、生成层和治理层
4. 当前项目里的 demo 是一个最小闭环，非常适合讲清原理
5. 从 demo 到生产级系统，关键是工程化、评测、治理和平台能力

### 讲解提示

收尾时不要再扩展新概念，把整场分享拉回这 5 句话即可。
