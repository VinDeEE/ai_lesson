## 2. RAG 是什么

RAG，`Retrieval-Augmented Generation`，中文通常叫“检索增强生成”。

最通俗的解释是：

**不让模型只靠参数记忆回答，而是让模型先去知识库里找证据，再基于证据组织答案。**

所以 RAG 的本质不是“让模型更聪明”，而是让模型回答时：

- 有外部依据
- 有最新知识
- 有引用来源
- 更容易控制幻觉

一句话区分：

- 纯大模型：靠参数记忆回答
- RAG：靠“外部证据 + 模型生成”回答

## 3. 为什么企业不能只靠纯大模型

在企业场景里，纯大模型通常很难直接满足要求，原因主要有 5 个：

1. 知识不完整  
   通用模型训练数据里没有企业内部 FAQ、SOP、制度、工单规则、产品文档等私域知识。

2. 知识更新滞后  
   企业知识持续变化，但模型参数不会自动更新；靠重新训练成本高、周期长。

3. 幻觉风险不可接受  
   模型可能给出“听起来合理但事实错误”的答案，而业务场景往往要求稳定、准确、可追责。

4. 数据安全要求更高  
   企业知识往往涉及权限、隐私、合规、审计，不适合直接暴露给通用外部模型。

5. 答案需要可追溯  
   业务回答通常不仅要“像是对的”，还要知道“依据是什么”，便于复核、审计和人工接管。

所以企业真正需要的，不是一个“会聊天”的模型，而是一个：

**能使用企业知识、能给出依据、能受控、能治理的知识问答系统。**

## 4. RAG 的价值和适用边界

### 4.1 RAG 的价值

RAG 的核心思路是：

**先从企业知识库中检索相关内容，再基于检索结果让大模型生成答案。**

它的价值主要体现在：

- 把知识更新成本从“重训模型”降到“更新知识库”
- 让模型具备回答企业内部问题的能力
- 降低幻觉概率，让回答更有依据
- 支持引用、溯源和审计
- 特别适合客服、知识库问答、坐席辅助、工单辅助、内部问答、运维助手等场景

### 4.2 适合 RAG 的场景

- FAQ、知识库问答
- 企业制度、流程、SOP 查询
- 客服辅助、工单辅助
- 产品文档、研发知识、运维知识问答
- 需要给出出处、证据或引用的回答场景

### 4.3 不一定适合 RAG 的场景

- 纯创作类内容
- 完全不依赖外部知识的开放式聊天
- 极短、极固定、完全可模板化返回的问答
- 对时延极端敏感且知识更新很少的场景

### 4.4 RAG 与微调不是替代关系

需要明确一点：

- 微调主要解决“能力风格、任务习惯、输出方式”
- RAG 主要解决“知识来源、知识时效、知识可追溯”

很多企业最终落地形态其实是：

**基础模型 + RAG + 少量任务级微调**

---

## 5. RAG 是怎么一步步发展到今天的

RAG 框架的发展大致经历了这样一条路线：

`Retrieve-and-Read -> 模型型 RAG -> 应用框架型 RAG -> Pipeline / Workflow 型 RAG -> Agentic RAG -> 企业平台化 RAG`

这条演进线很重要，因为它解释了为什么今天市面上的“RAG 框架”看起来差别很大。

### 5.1 前史：Retrieve-and-Read

在 `2020 年之前`，业界已经有“先检索再阅读”的问答系统思路，但还没有今天这么完整的 RAG 框架生态。

### 5.2 模型型 RAG

`2020 年` 的原始 RAG 论文把“检索增强生成”作为一个明确方法建立起来。  
这一阶段重点是模型范式，不是业务系统。

### 5.3 应用框架型 RAG

`2023 年前后`，随着大模型应用爆发，开始出现大量帮助开发者快速搭建 RAG 的框架，如：

- LangChain
- LlamaIndex
- 早期 Haystack RAG 方案

这一阶段的典型抽象开始稳定为：

- 文档加载
- 文本切块
- 向量化
- 索引/向量库
- 检索
- Prompt 编排
- LLM 生成

### 5.4 Pipeline / Workflow 型 RAG

`2024 年前后`，大家发现简单的 `retrieve -> generate` 不够用，开始加入：

- rerank
- filter
- fallback
- output validation
- handoff
- 评测与观测

于是 RAG 从“一条链”逐步变成“可编排的工程流水线”。

### 5.5 Agentic RAG

`2025 年前后`，越来越多框架开始把检索变成 Agent 可动态调用的能力。

这意味着：

- 检索次数不再固定
- 会做 query rewrite
- 会拆解子问题
- 会动态决定什么时候继续查、什么时候停止查

### 5.6 企业平台化 RAG

再往前走，RAG 不再只是“问答链路”，而是“知识平台”：

- 多数据源接入
- 文档解析与清洗
- 增量更新
- 混合检索与重排
- 权限控制
- 引用审计
- 监控、成本和运维治理

这时，RAG 的重点已经不只是模型，而是平台能力。

## 6. 一个企业级 RAG 系统完整长什么样

企业级 RAG 不只是“向量检索 + 大模型”，至少要包含 4 层能力：

1. 数据层
2. 检索层
3. 生成与编排层
4. 评测与治理层

### 6.1 总体架构

可以用下面这条主链来讲：

```text
数据源
-> 文档接入 / 解析 / 清洗
-> 切块 Chunking
-> Embedding
-> 向量库 / 索引

用户问题
-> Query Embedding
-> Retrieve
-> Rerank / Filter
-> Prompt / Workflow
-> LLM
-> Answer / Citation / Handoff

评测、监控、权限、审计、缓存、成本治理
贯穿整个系统
```

配图建议：`插图/企业级rag架构.png`

### 6.2 离线阶段做什么

离线阶段的目标是：

**把原始文档变成可检索的知识资产。**

典型步骤包括：

- 接入 FAQ、SOP、产品文档、制度文档、工单知识、网页、PDF 等数据源
- 清洗脏数据、去重、结构化
- 进行合理切块
- 为 chunk 生成 embedding
- 写入向量索引或混合索引
- 保存 metadata，如来源、版本、更新时间、权限标签

### 6.3 在线阶段做什么

在线阶段的目标是：

**把用户问题变成一个“有证据约束的答案”。**

典型步骤包括：

- 理解用户问题
- 把问题转成向量
- 检索相关证据
- 必要时做 rerank 和过滤
- 把证据组织成 prompt
- 让大模型生成答案
- 输出 citation、confidence、need_handoff 等控制字段

### 6.4 这一部分分享时要强调什么

这一页最值得强调的不是“调用了哪个模型”，而是：

- 数据质量比模型大小更重要
- 切块策略直接影响召回质量
- 检索质量直接决定答案上限
- 生成不是凭空回答，而是对证据的组织和压缩
- 企业级落地必须考虑评测、权限、监控和治理

## 7. 市面上的主流 RAG 框架，大致分哪几类

现在常见的 RAG 框架，不是同一层东西，最好分成 3 类：

### 7.1 代码框架

代表：

- LangChain / LangGraph
- LlamaIndex
- Haystack

特点：

- 灵活度高
- 适合自己写工程
- 适合复杂编排、复杂检索、复杂工作流

### 7.2 低代码 / 可视化平台

代表：

- Dify
- Flowise

特点：

- 拖拽式工作流
- 适合快速做知识库问答、PoC 和业务应用
- 展示性强，交付快

### 7.3 一体化平台 / 托管平台

代表：

- RAGFlow
- Pathway
- Azure AI Search
- Bedrock Knowledge Bases
- Vertex AI RAG Engine

特点：

- 更关注数据接入、索引、权限、评测、监控、平台能力
- 更像企业级知识平台，而不只是一个编排库

这一页分享时可以直接给听众一个结论：

**今天大家都在做 RAG，但差异不在“能不能检索增强”，而在“它把哪一层做成了自己的优势”。**

## 8. 回到当前项目：一个最小可运行的 RAG demo

当前项目里的 demo 可以定义为：

**一套“本地 JSON 索引 + Gemini Embedding + Gemini Generate”的最小可用 RAG 实现。**

它非常适合做分享，因为：

- 链路清晰
- 每一步都透明
- 没有复杂框架把底层逻辑遮住
- 能很直观地展示 RAG 的最小闭环

### 8.1 当前 demo 的核心组件

- 文档源：`docs/` 下的 FAQ、SOP 等文本
- 入库脚本：`src/ingest.py`
- 预设入库入口：`src/week3_ingest_runner.py`
- 索引文件：`data/chroma/kb.json`
- 问答入口：`src/ask.py`
- 核心引擎：`src/rag_core.py`
- 批量评测：`src/evaluate.py`
- 向量模型：Gemini Embedding
- 生成模型：Gemini Generate

### 8.2 当前 demo 真实做了哪 3 件事

1. 离线入库
2. 在线问答
3. 批量评测

### 8.3 它最关键的依赖关系

```text
docs/*
  -> ingest.py
  -> data/chroma/kb.json
  -> ask.py / evaluate.py
```

也就是说：

- `ask.py` 和 `evaluate.py` 不直接读取 `docs/`
- 它们依赖的是预先构建好的 `kb.json`
- 只有先完成入库，问答和评测链路才能跑起来

### 8.4 当前 demo 的本质定位

这里一定要讲清楚：

- 当前索引虽然目录名叫 `data/chroma/`
- 但当前并没有真正使用 Chroma 向量数据库
- 当前检索不是 ANN
- 当前是本地 JSON 索引 + Python 全量相似度扫描

所以它的真实定位不是“企业级向量检索平台”，而是：

**一个教学型、最小闭环、便于讲清 RAG 原理的 demo。**

## 9. 当前 demo 的完整流程是什么

### 9.1 离线入库链路

离线入库的作用是：把文档变成可检索知识。

实际流程可以概括为：

```text
读取 docs/
-> split_text 切块
-> 调用 Gemini Embedding 生成 chunk 向量
-> 把 text + metadata + embedding 保存进 kb.json
```

### 9.2 在线问答链路

在线问答的作用是：把用户问题变成结构化答案。

实际流程可以概括为：

```text
用户问题
-> ask.py
-> answer_question()
-> load_index(kb.json)
-> embed_texts([question])
-> 本地遍历全部 chunk 做 cosine similarity
-> 取 top_k 证据
-> 组装 context / prompt
-> 调用 Gemini Generate
-> 输出 answer / citations / confidence / need_handoff
```

### 9.3 哪些逻辑在本地，哪些在外部

外部服务只做两类事情：

- Gemini Embedding：文档向量化、问题向量化
- Gemini Generate：基于证据生成最终答案

本地逻辑包括：

- 读取 `.env`
- 扫描文档
- 切块
- 保存和加载 `kb.json`
- 本地相似度排序
- 拼装 prompt
- 解析结构化输出
- 构造 citations / confidence / need_handoff
- 做简单回归评测

### 9.4 这一页最值得讲的点

你要让听众明确看到：

- embedding 解决的是“怎么比较相似”
- retrieval 解决的是“找到什么”
- generation 解决的是“怎么回答”

## 10. 为什么这个 demo 要保留“有模型”和“无模型”两种模式

当前 demo 支持两种问答模式：

1. 默认模式  
   先检索，再调用生成模型输出结构化答案

2. `--vector_only` 模式  
   只做向量检索，不调用生成模型，直接返回召回片段

这页在分享里非常有价值，因为它能非常直观地回答一个问题：

**既然向量已经能找到相关证据，为什么还要再调一次大模型？**

答案是：

- 向量检索解决的是“找到什么”
- 大模型生成解决的是“怎么组织成可读、可执行、可交付的答案”

所以：

- 无模型模式更适合展示检索层
- 有模型模式更适合展示完整 RAG 闭环

一句话结论：

**检索负责召回，大模型负责表达。**

## 11. 当前 demo 的优点与边界

### 11.1 当前 demo 的优点

- 链路清晰，适合教学和分享
- 数据结构可见，便于理解
- 依赖少，容易调试
- 已经具备最小闭环：入库、问答、评测
- 能通过 `--vector_only` 明确拆开检索层和生成层

### 11.2 当前 demo 的边界

- 还不是真正的向量数据库方案
- 检索还是全量扫描，不是 ANN
- 没有 BM25 / sparse retrieval
- 没有 hybrid retrieval
- 没有 rerank
- 没有权限治理
- 评测还比较轻
- 离企业平台化方案还有距离

这一页的意义在于：

**既说明 demo 已经具备教学价值，也说明它为什么还不是生产级系统。**

## 12. 从 demo 走向工程化和生产化，需要补哪些能力

### 12.1 检索层增强

- 向量库或 ANN 索引
- BM25 / sparse retrieval
- hybrid retrieval
- rerank
- query rewrite
- metadata filter

### 12.2 数据层增强

- 更严格的数据边界管理
- 业务知识与学习笔记分离
- 增量入库
- 去重
- 更丰富的 metadata
- 文档版本和更新时间管理

### 12.3 生成层增强

- 更严格的 prompt 约束
- 更稳定的 JSON schema
- 输出校验
- need_handoff 策略优化
- 更细粒度的引用控制

### 12.4 评测层增强

- Recall@K
- MRR / nDCG
- Faithfulness
- Citation Precision
- 人工标注集
- 回归评测体系

### 12.5 平台层增强

- 权限控制
- 多知识库
- 模型路由
- 缓存
- 监控告警
- 成本统计
- 调用链追踪

一句话说，这一部分的核心不是“继续堆功能”，而是：

**把 demo 逐步推进成可治理、可观测、可扩展、可维护的工程系统。**

## 14. 分享时建议的讲法

### 14.1 开头

不要一上来就堆术语，建议从一个业务问题切入。

例如：

> 用户问：“工单时效有问题。”  
> 如果只靠大模型，它可能会给出一个看似合理但没有依据的答案。  
> 如果接入 RAG，系统就会先去企业知识库里找 FAQ、SOP、工单规则，再基于证据生成答案。

### 14.2 中段

中段先讲企业级完整架构，再落到当前 demo。

推荐顺序：

1. RAG 的概念和价值
2. 企业为什么需要 RAG
3. 企业级完整架构和数据流
4. 当前 demo 的真实实现
5. 当前 demo 的边界
6. 工程化扩展方向

### 14.3 落到项目

你要让听众清楚：

- 当前 demo 不是 PPT 概念图，而是真能跑起来的闭环
- 它已经具备入库、问答、评测能力
- 它的价值不在“有多复杂”，而在“把最核心的链路跑通了”

### 14.4 结尾

最后建议用下面这段话收束：

> RAG 的本质，不是给大模型简单加一个检索步骤，而是把企业知识、检索系统、生成模型、评测体系和工程治理整合成一套可落地的方法。  
> 当前项目里的 demo 已经把最核心的闭环跑通了。下一步真正要做的，不只是继续堆功能，而是围绕数据治理、检索质量、评测体系、权限控制、成本与运维，把它逐步推进成工程化方案。

## 15. 一页总结

如果要把整场分享压缩成 5 句话，可以直接用下面这版：

1. RAG 的本质是“先找证据，再组织答案”
2. 企业需要 RAG，是因为纯大模型缺少私域知识、时效性、可控性和可追溯性
3. 一个完整的 RAG 系统包含数据层、检索层、生成层和治理层
4. 当前项目里的 demo 是一个最小可运行的 RAG 闭环，非常适合讲清原理
5. 从 demo 到生产级系统，关键是工程化、评测、治理和平台能力
