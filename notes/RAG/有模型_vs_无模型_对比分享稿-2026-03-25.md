# 有模型 vs 无模型 对比分享稿

## 1. 这页要讲什么

这一页不是讲抽象理论，而是用当前项目里的真实输出，解释一个非常关键的问题：

**为什么 RAG 在检索到证据之后，仍然需要再调用一次大模型？**

结论先说：

- 仅向量模式可以把“相关片段”找出来
- 但大模型模式可以把“相关片段”变成“可读、可用、可总结的答案”

所以：

**向量检索解决的是“找到什么”，大模型生成解决的是“怎么回答”。**

---

## 2. 演示命令

```powershell
python src/ask.py "工单时效有问题"
python src/ask.py "工单时效有问题" --vector_only
```

说明：

- 第一条：正常 RAG 模式，先检索，再调用大模型生成答案
- 第二条：仅向量模式，只检索，不调用生成模型

## 3. 当前真实对比样例

测试问题：

```text
工单时效有问题
```

### 3.1 有模型模式输出

核心结果：

```text
客服系统-工单详情页的工单时效存在两个问题：
1. 首次打开工单时，工单时效显示乱码；
2. 切换页面后再回到当前工单，处理信息模块不显示工单时效。
其中，工单时效组件不显示的问题已在1月27号上线版本中修复。
```

...

## 4. 这组对比最值得讲的点

### 4.1 两种模式的检索其实是一样的

这次对比里，两种模式的检索元信息是相同的：

- `top_k = 4`
- `top_score = 0.7544604797441561`
- 第一条核心证据都来自 `faq.md | faq#2`

也就是说：

**差异不是出在“有没有找到证据”，而是出在“拿到证据之后怎么处理”。**

### 4.2 无模型模式的特点

优点：

- 完全透明，能看到原始召回内容
- 更容易讲清楚“RAG 的检索层到底做了什么”
- 适合排查召回是否正确

缺点：

- 结果里混入了无关片段
- 信息冗长，不适合直接给业务用户看
- 需要用户自己从原始片段里提炼结论

一句话概括：

**无模型模式更像“把相关材料扔给你”，但不会帮你整理答案。**

### 4.3 有模型模式的特点

优点：

- 从召回片段里抽取出真正和问题相关的要点
- 自动去掉无关噪音
- 语言更简洁，更适合直接展示给用户
- 可以进一步输出 `need_handoff`、`confidence`、`citations` 这类结构化结果

缺点：

- 多一次模型调用，增加时延和成本
- 如果 prompt 设计不好，可能误总结、漏总结

一句话概括：

**有模型模式不是重新查资料，而是把已经查到的资料“整理成答案”。**

## 5. 一张对比表

| 维度 | 仅向量模式 | 有模型模式 |
|---|---|---|
| 是否调用生成模型 | 否 | 是 |
| 检索能力 | 一样 | 一样 |
| 可读性 | 低 | 高 |
| 信息噪音 | 高 | 低 |
| 是否适合直接给用户看 | 一般不适合 | 适合 |
| 是否适合排查召回问题 | 很适合 | 一般 |
| 成本 | 更低 | 更高 |
| 分享时的展示价值 | 能讲清检索层 | 能讲清完整 RAG 价值 |

---

## 6. 分享时建议怎么讲

可以直接这样说：

> 我们特意加了一个 `--vector_only` 模式，目的不是替代大模型，而是把“检索层”和“生成层”拆开看。  
> 当我们只用向量检索时，系统确实能把相关片段找出来，但它返回的是原始材料，不是整理好的答案。  
> 当我们加上大模型后，系统不会改变检索结果本身，而是会基于这些证据做压缩、归纳和表达优化。  
> 所以在这个 demo 里，大模型的价值不是“再去搜索一次”，而是“把检索结果变成用户真正能用的回答”。

---

## 7. 最适合落地到 PPT 的一句话

**仅向量检索回答的是“我找到了哪些相关片段”，大模型回答的是“基于这些片段，我给你一个可直接使用的答案”。**

---

## 8. 如果现场有人追问

### 问：既然向量已经能找到 `faq#2`，那为什么不能直接结束？

可以这样答：

- 因为业务用户通常不想看原始片段
- 用户要的是“整理后的结论”
- 检索结果里常常混有噪音，需要做筛选和总结
- 大模型正是负责这一步

### 问：那是不是所有场景都必须用大模型？

可以这样答：

- 不一定
- FAQ、固定模板场景，有时只检索也够
- 但一旦涉及总结、改写、跨片段合并、结构化输出，还是需要生成模型

---

## 9. 结尾总结

这组对比最重要的价值，不是证明“大模型一定更好”，而是帮听众看清：

1. 检索层和生成层是两件不同的事
2. 向量检索负责召回
3. 大模型负责压缩、整理、表达
4. 真正可用的 RAG，通常不是“只检索”或“只生成”，而是两者协同
