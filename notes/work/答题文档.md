# Round 4 答题文档

以下答案按“客服系统 + AI 推荐答案 + 知识库 + 转人工”这条项目线来组织，目标不是讲概念，而是把你的职责、设计思路、指标口径和技术判断说清楚。

## 1. 企业知识库是怎么建设和更新的，谁来维护，怎么保证内容有效。

这个问题我一般会从“来源、维护、更新、治理”四层来答。

我们当时的企业知识库不是一次性整理出来的，而是从几个稳定业务源逐步沉淀的，主要包括历史工单、客服标准话术、产品/运营规则文档、常见售后问题，以及一线客服日常处理过程中沉淀下来的高频问答。我的做法不是把所有资料直接丢给模型，而是先做结构化治理，按业务域拆成不同知识分类，比如寄件、取件、费用、异常件、投诉、活动规则等，再给每条知识打上场景标签、适用条件、时效信息和版本信息，这样后面检索和推荐才可控。

维护上不是单纯由技术团队维护，而是采用“业务 owner + 客服主管 + 技术平台”三方协同。业务 owner 负责规则正确性，客服主管负责内容是否符合一线使用习惯，技术侧负责知识入库、版本管理、发布流程和效果监控。这样可以避免知识库变成技术自己维护、但业务不买单的情况。

更新机制上，我们做的是“日常增量更新 + 重点变更强制更新”两套机制。日常增量主要来自高频新问题、转人工原因分析和低命中会话复盘；重点变更则来自业务规则调整、活动上线、产品流程变化、投诉案例升级，这类内容会走加急发布。我的理解是知识库更新不能靠人工想起来再改，而要和业务变更流程绑定，否则一定会滞后。

怎么保证内容有效，我主要抓 4 件事：

1. 准入机制：不是所有内容都能直接进知识库，必须经过业务确认，避免把历史错误答案继续放大。
2. 版本和时效：知识要有生效时间、失效时间和版本号，尤其是活动规则、费用规则这类强时效内容。
3. 效果回流：重点看知识命中率、推荐后采纳率、命中后是否仍转人工，以及错误答案的复盘原因。
4. 定期清理：对低命中、已过期、重复冲突内容做下线和合并，保证知识库不是越堆越乱。

如果面试官追问一句“你本人具体负责什么”，可以补一句：

我本人主要负责知识库体系的建设方法和治理机制，包括分类规则、入库标准、更新流程、效果监控口径，以及推动业务和客服团队形成长期维护机制，而不是只负责技术接入。

## 2. AI 推荐答案这条链路里，哪些是规则，哪些是模型，哪些是检索。

这个问题一定要把链路拆开回答，不然很容易被认为只是把“大模型 + 知识库”几个词堆在一起。

我们的链路大致是“用户问题进入 -> 做意图识别和基础预处理 -> 检索候选知识 -> 结合规则过滤和排序 -> 生成或推荐答案 -> 判断是否转人工”。

这里面规则、模型、检索的分工是这样的：

规则层主要做确定性控制，负责兜底和风控。比如敏感词识别、投诉类场景识别、身份校验前置条件、活动/费用类强规则限制、是否必须转人工的场景判断、答案发布权限控制等，这些我不会交给模型自由发挥，因为这类问题一旦答错，业务风险很高。

模型层主要做语义理解和答案组织。比如用户问题改写、意图识别、多轮上下文理解、相似问题归一、候选答案排序，以及在低风险场景下对知识内容做自然语言重组，让回复更接近人工客服表达。简单说，模型负责“理解”和“表达”，但不单独决定高风险业务规则。

检索层主要负责从知识库里召回候选内容。我们会按照用户问题、上下文、业务标签去匹配知识条目，先召回一批候选，再结合规则和模型做二次排序。检索解决的是“从哪里找答案”，模型解决的是“用户到底在问什么、怎么把答案讲清楚”，规则解决的是“哪些答案能说、哪些场景不能自动说”。

如果要再压缩成一句面试里的总结，我会这么说：

我的设计原则是，规则负责边界，检索负责找知识，模型负责理解和表达，三者分层而不是互相替代。这样系统才既有效率，也可控。

## 3. 你们怎么处理幻觉、错答、敏感回复和高风险场景。

这个问题本质上考的是你有没有线上风险意识。我的回答思路是“先限制、再判断、后兜底”。

第一层是知识边界限制。模型不能脱离知识库自由回答，优先基于已审核知识生成或推荐内容。如果没有命中可靠知识，宁可降级成引导澄清、推荐人工，也不让模型编造。

第二层是场景分级。我们把场景分成低风险、中风险、高风险。像一般咨询、流程说明、常见问答属于低风险，可以自动推荐；涉及费用争议、赔付、投诉升级、账户异常、政策边界解释等属于中高风险，这类要么走强规则模板，要么直接转人工，不能让模型自由组织。

第三层是敏感内容拦截。对辱骂、投诉、法律风险、舆情风险、个人隐私等内容，会先经过规则和关键词/分类模型双重判断。一旦命中敏感场景，就切换到固定回复、安抚模板或人工介入流程，避免答非所问或者激化用户情绪。

第四层是结果校验。即使召回了知识，也不是直接发给用户。我们会做答案置信度判断、知识一致性校验，以及必要的字段校验。比如费用、时效、活动资格这种容易出错的信息，必须来自明确规则或标准知识，不允许模型改写出新口径。

第五层是事后复盘。线上一旦出现错答、低采纳、重复转人工，我们会回看是知识问题、检索问题、模型理解问题，还是规则漏拦截，再对应修知识、调召回、补规则，而不是笼统说模型效果不好。

如果面试官追问“你们怎么定义幻觉”，可以直接说：

在我们场景里，幻觉不只是模型胡编乱造，还包括答非所问、知识过期还在答、规则类问题给出了不符合当前业务口径的回复。这三类在线上都按高优先级问题处理。

## 4. 你们怎么平衡效果、响应时间和成本。

这个问题不要回答成空泛的“三者权衡”，要讲具体方法。

我的思路是先分场景，再分层路由，而不是所有请求都走同一套最重的链路。高频、标准化、低风险问题，优先走规则 + 检索 + 模板化回复，这类响应快、成本低，而且稳定。只有在用户表达不标准、需要语义理解、或者需要多轮上下文判断时，才让模型参与更深一层处理。

在效果上，我们看的是用户问题是否被真正解决，所以不会只看模型命中率，还会看推荐采纳率、转人工率、人工介入后的修正率、用户满意度这些指标。也就是说，如果某条链路看起来回答很多，但最后还是大量转人工，那它并不算真正有效。

在响应时间上，我会控制链路层级和超时策略。比如先快速走意图判断和候选召回，超过阈值就降级，不会为了追求更高一点点效果让用户一直等。客服场景里响应时延本身就是体验的一部分，尤其在线会话里，慢回复会直接降低用户信任。

在成本上，我们会把模型使用放在真正需要它的地方，而不是全链路默认调用；同时通过高频问题知识化、模板化、缓存化，尽量把可标准化的问题前置解决。我的原则是，能用规则稳定解决的，不强上模型；能用检索解决的，不强做生成；只有规则和检索都不足时，才让模型发挥价值。

如果面试时需要一句总结，我会这样收口：

我平衡这三件事的方法不是单点优化，而是做分层路由。低成本链路吃掉大部分标准问题，把高成本能力留给真正复杂的问题，这样整体效果、时延和成本才同时成立。

## 5. 你亲自拍板过的一个关键技术决策是什么，为什么这么定。

这个问题建议不要回答得太虚，最好讲一个“有争议、你做了取舍、最后结果验证有效”的决策。

我会答这个：

我亲自拍板过的一个关键技术决策，是没有把客服问答完全做成“大模型直接生成答案”，而是坚持做“检索增强 + 规则兜底 + 分场景路由”的架构。

当时之所以这么定，核心有 3 个原因。

第一，客服场景对正确性和一致性要求很高，尤其涉及费用、规则、投诉、异常件这些内容，如果完全交给模型自由生成，短期看体验可能更自然，但线上风险太大，而且很难解释为什么答成这样。

第二，我们的知识和业务规则本身变化频繁，如果只依赖模型参数，不仅更新慢，也很难做到版本可追踪。做成知识检索和规则配置后，业务一变更就可以快速更新，不需要等模型重新训练或重新调参。

第三，系统上线不是只比效果，还要比可维护性和可运营性。分层架构虽然前期设计更重一些，但后面出了问题可以快速定位到底是知识问题、检索问题、模型问题还是规则问题，方便持续迭代。

这项决策带来的价值是两方面：一方面让系统在上线初期更稳，能控制风险；另一方面也让后续优化路径非常清楚，我们可以分别优化知识质量、召回排序、模型理解和转人工策略，而不是所有问题都归因给模型。

如果面试官继续追问“为什么这是你拍板的，而不是团队自然形成的”，你可以补一句：

因为这个决策本质上决定了项目是走“效果优先的模型方案”，还是走“可控优先的工程方案”。这不是单点实现选择，而是整体架构方向，我当时是基于业务风险、系统可维护性和落地节奏综合判断后定下来的。

## 建议你现场回答时的节奏

这 5 个问题都不要答得太散，建议统一按这个逻辑来：

1. 先说结论和你的判断。
2. 再拆成 2 到 4 个关键点。
3. 最后补一句“我本人负责什么”或“为什么这么设计”。

比如你每题都尽量落到这几个词上：

- 我们不是简单接模型，而是做了分层设计。
- 我负责的是机制设计和落地推进，不只是参与开发。
- 我们重点解决的是正确性、可控性、时效性和持续优化。
- 线上系统的核心不是回答得像人，而是答得对、答得稳、能持续迭代。

## 一版更像面试口语的精简回答

如果现场时间很短，可以直接按下面的短版说：

### 1. 知识库怎么建和更新

我们的知识库主要来自历史工单、标准话术、产品规则和一线客服经验沉淀。建设时我先推动做业务分类、标签、版本和时效治理，不是把文档直接丢给模型。维护上是业务 owner、客服主管和技术平台一起负责，业务保正确性，客服保可用性，技术保入库和发布流程。有效性主要靠准入审核、版本管理、效果回流和定期清理来保证。

### 2. 哪些是规则、模型、检索

我把这条链路分成三层：规则负责边界和风控，比如敏感场景、必须转人工场景；检索负责从知识库召回候选答案；模型负责语义理解、上下文判断和答案组织。简单说，规则管能不能答，检索管去哪里找，模型管怎么理解和怎么说。

### 3. 怎么处理幻觉和高风险场景

我们的原则是模型不能脱离已审核知识自由发挥，没有可靠知识就降级或者转人工。再加上场景分级，把费用、赔付、投诉、账户异常这类高风险场景单独拦出来，走规则模板或人工接管。对敏感内容会做拦截和固定回复，线上错答会回溯到底是知识、检索还是规则问题，再针对性修正。

### 4. 怎么平衡效果、响应时间和成本

我不是让所有请求都走最重的模型链路，而是按场景做分层路由。高频标准问题优先走规则和检索，复杂问题再调用模型。这样既保证大部分请求响应快、成本低，也把模型资源留给真正需要语义理解的问题。最终评估不是只看命中率，而是看采纳率、转人工率和用户体验。

### 5. 一个关键技术决策

我亲自拍板的关键决策，是没有做纯大模型生成，而是做检索增强、规则兜底和分场景路由。因为客服场景更看重正确性、可控性和可维护性，这样设计虽然前期复杂一点，但后续好治理、好扩展、风险也更可控。这也是系统能稳定上线并持续优化的关键。
